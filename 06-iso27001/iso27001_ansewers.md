# ISO 27001 Annex A.8 — Контроли A.8.1 – A.8.5

## Таблица соответствия

| Контроль | Название | Реализовано? | Как именно / Почему нет |
|----------|---------|:---:|------------------------|
| **A.8.1** | User endpoint devices | Частично | Сервер захарденен в задании 01: закрыт файрвол UFW (открыты только SSH 2222 и веб 8080), отключены ненужные сервисы (cups, avahi, bluetooth), включён auditd для логов. Но полная защита endpoint-устройств (антивирус, шифрование дисков, MDM, политика для ноутбуков разработчиков) — не делалась, потому что задание про сервер, а не про рабочие станции. |
| **A.8.2** | Privileged access rights | Да | В задании 01 создан обычный пользователь `dev` с ограниченным sudo — может запускать только systemctl, docker, journalctl, apt-get (не полный root). В SSH запрещён вход под root. В задании 02 в Vault создана политика `app-readonly` — приложение может только читать секреты, но не менять и не удалять. В задании 05 у GitHub Actions минимальные permissions. |
| **A.8.3** | Information access restriction | Да | В задании 02 Vault даёт доступ к секретам только по политикам — разные токены получают разные права. В задании 03 секреты убраны из Docker-образа (передаются через переменные окружения при запуске). В задании 01 файрвол UFW работает по принципу deny-by-default — всё запрещено кроме явно разрешённых портов. |
| **A.8.4** | Access to source code | Частично | В задании 04 настроен gitleaks + pre-commit хук, который блокирует коммит если находит секреты в коде. В задании 05 в CI/CD pipeline тоже есть gitleaks — проверяет каждый PR. В задании 03 `.dockerignore` не пускает папку `.git` в Docker-образ (там могут быть старые секреты в истории). Но branch protection rules, обязательный code review, подпись коммитов через GPG, CODEOWNERS — не настраивались, это нужно делать в настройках самого GitHub/GitLab. |
| **A.8.5** | Secure authentication | Да | В задании 01 SSH настроен на вход только по ключам (пароли запрещены), `MaxAuthTries=3`, установлен fail2ban — банит IP после 3 неудачных попыток. В задании 02 Vault использует токены с ограниченным сроком жизни (TTL) — токен перестанет работать автоматически через час, даже если утечёт. Root-токен используется только для начальной настройки. |

## Какой контроль самый важный для DevSecOps?

**A.8.2 — Privileged access rights (управление привилегированным доступом).**

DevSecOps каждый день работает с системами где привилегированный доступ есть везде: CI/CD пайплайны с секретами для деплоя, Docker где по умолчанию всё от root, облачные IAM-роли, SSH-ключи к продакшен-серверам. Если не контролировать кто какие права имеет — все остальные меры безопасности теряют смысл. Зачем хранить пароли в Vault, если у каждого разработчика есть root-токен? Зачем настраивать файрвол, если все процессы работают под root внутри контейнера?

Принцип наименьших привилегий (least privilege) проходит через все 5 заданий этой работы — ограниченный sudo у пользователя dev, read-only политика в Vault, непривилегированный юзер в Dockerfile, минимальные `permissions` в GitHub Actions. A.8.2 — это тот контроль который объединяет все остальные в единую систему и без которого остальные не работают.
